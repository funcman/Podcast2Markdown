# 关于桌面端化的调研

**调研日期**：2025-04-03  
**调研目标**：评估将 Podcast2Markdown 改造为 Windows 桌面应用的难度和方案

---

## 一、当前项目组件分析

| 组件 | 技术栈 | 桌面化难度 | 说明 |
|------|--------|-----------|------|
| 前端界面 | Next.js + React + Tailwind CSS | ⭐ 容易 | Web 技术，可直接复用 |
| 后端 API | Next.js API Routes (Node.js) | ⭐ 容易 | Electron 可直接运行 Node.js |
| 数据库 | SQLite + Prisma | ⭐ 容易 | SQLite 本身就是本地数据库 |
| 音频转码 | fluent-ffmpeg + 系统 ffmpeg | ⚠️ 中等 | 需打包或引导安装 ffmpeg |
| Whisper 转录 | whisper.cpp (子进程 spawn) | ⭐ 容易 | 已经是可执行文件，直接调用 |
| AI 文章生成 | Minimax API | ⭐ 容易 | HTTP 请求，仅需网络 |

**整体评估**：现有架构非常适合桌面化改造，90% 代码可直接复用。

---

## 二、改造方案对比

### 方案 1：Electron（推荐 ⭐⭐⭐⭐⭐）

**难度**：★☆☆☆☆（最小）  
**预计工作量**：1-2 天

#### 工作原理
```
┌─────────────────────────────────────────┐
│           Electron 主进程              │
│  ┌─────────────────────────────────┐   │
│  │  Node.js 运行时                  │   │
│  │  ┌─────────┐  ┌──────────────┐ │   │
│  │  │ Next.js │  │ whisper.cpp  │ │   │
│  │  │ API     │  │ (子进程)      │ │   │
│  │  │ Routes  │  │              │ │   │
│  │  └────┬────┘  └──────────────┘ │   │
│  │       │                        │   │
│  │  ┌────▼────┐  ┌──────────────┐ │   │
│  │  │ SQLite  │  │ fluent-ffmpeg│ │   │
│  │  │ (Prisma)│  │ → 调用 ffmpeg│ │   │
│  │  └─────────┘  └──────────────┘ │   │
│  └─────────────────────────────────┘   │
│                   │                     │
│  ┌────────────────▼─────────────────┐  │
│  │   BrowserWindow (Chromium)       │  │
│  │   ┌───────────────────────────┐  │  │
│  │   │  Next.js 前端页面 (复用)   │  │  │
│  │   │  上传组件、进度条、结果展示 │  │  │
│  │   └───────────────────────────┘  │  │
│  └──────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

#### 改造步骤
1. `npm install @electron/remote electron electron-builder`
2. 创建 `electron/main.js` 入口文件
3. 启动 Next.js 开发服务器（开发模式）或加载打包文件（生产模式）
4. 配置 `electron-builder` 打包脚本
5. 测试 whisper.cpp 和 ffmpeg 调用路径

#### 优点
- ✅ 100% 复用现有代码，零重构
- ✅ Node.js 主进程直接运行后端逻辑
- ✅ whisper.cpp、ffmpeg 调用方式完全不变
- ✅ 生态成熟，文档丰富，社区支持好
- ✅ 跨平台（Windows、macOS、Linux）

#### 缺点
- ⚠️ 包体积较大（约 150-200MB，含 Chromium）
- ⚠️ 内存占用比原生应用高

---

### 方案 2：Tauri（推荐 ⭐⭐⭐）

**难度**：★★★☆☆（中等）  
**预计工作量**：1-2 周

#### 工作原理
```
┌─────────────────────────────────────────┐
│           Tauri 主进程                  │
│  ┌─────────────────────────────────┐   │
│  │  Rust 运行时                     │   │
│  │  ┌─────────┐  ┌──────────────┐ │   │
│  │  │ 重写 API│  │ 绑定 whisper │ │   │
│  │  │ 路由    │  │ (Rust FFI)   │ │   │
│  │  │ (Rust)  │  │              │ │   │
│  │  └────┬────┘  └──────────────┘ │   │
│  │       │                        │   │
│  │  ┌────▼────┐  ┌──────────────┐ │   │
│  │  │ SQLite  │  │ 绑定 ffmpeg  │ │   │
│  │  │ (Rust)  │  │ (Rust FFI)   │ │   │
│  │  └─────────┘  └──────────────┘ │   │
│  └─────────────────────────────────┘   │
│                   │                     │
│  ┌────────────────▼─────────────────┐  │
│  │   WebView2 (Windows) /           │  │
│  │   WKWebView (macOS)              │  │
│  │   ┌───────────────────────────┐  │  │
│  │   │ 前端页面 (需适配)          │  │  │
│  │   │ 可用 React/Vue/Vanilla    │  │  │
│  │   └───────────────────────────┘  │  │
│  └──────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

#### 改造步骤
1. 安装 Rust 工具链
2. `npm create tauri-app` 创建新项目
3. **用 Rust 重写所有后端逻辑**：
   - API 路由（需使用 Rust Web 框架如 Axum）
   - whisper.cpp 调用（使用 `std::process::Command`）
   - ffmpeg 调用（同上）
   - SQLite 操作（使用 `rusqlite` 或 `sqlx`）
4. 前端可选保持 React 或重写为更轻量方案
5. 配置 Tauri 打包

#### 优点
- ✅ 包体积小（约 10-20MB）
- ✅ 使用系统 WebView（Windows 用 Edge WebView2）
- ✅ Rust 性能更好，内存占用低
- ✅ 更原生 Windows 体验

#### 缺点
- ❌ **需要学习 Rust 语言**
- ❌ 需要重写所有后端逻辑（约 1000+ 行代码）
- ❌ 调试和学习曲线较陡
- ❌ 社区资源和文档比 Electron 少

---

### 方案 3：Neutralinojs

**难度**：★★☆☆☆  
**预计工作量**：3-5 天

#### 特点
- 比 Electron 轻量，但功能也相对简单
- 使用系统 WebView
- 需要额外处理 native module 的加载路径问题

#### 优点
- ✅ 包体积中等（约 30-50MB）
- ✅ 内存占用低于 Electron

#### 缺点
- ⚠️ 社区较小，生态不成熟
- ⚠️ 文档相对较少
- ⚠️ 某些复杂场景支持不如 Electron

## 四、需要额外考虑的问题

### 1. ffmpeg 分发方案

| 方案 | 实现方式 | 优缺点 |
|------|----------|--------|
| **打包进应用** | 将 ffmpeg.exe 放入 resources/ | ✅ 用户无需安装<br>⚠️ 增加 50-100MB 包体积 |
| **引导用户安装** | 检测 → 提示下载 → 配置路径 | ✅ 包体积小<br>⚠️ 用户体验稍差 |
| **首次运行时下载** | 检测 → 自动下载到应用目录 | ✅ 包体积小<br>⚠️ 需处理下载失败和网络问题 |

**推荐**：首次运行时自动下载（平衡体积和体验）

---

### 2. Whisper 模型分发

| 方案 | 说明 |
|------|------|
| **引导用户下载** | 首次运行提示下载模型（3GB） |
| **分版本发布** | 提供带模型版（大）和不带模型版（小） |
| **增量更新** | 使用差量更新机制 |

**推荐**：引导用户下载，避免初始包过大

---

### 3. API Key 配置

桌面应用需要提供配置界面：
```
设置面板
├── Minimax API Key 输入框
├── 模型路径选择
├── ffmpeg 路径配置（可选）
└── 主题/语言偏好
```

配置存储：
- Electron：`electron-store` 或 SQLite
- 加密存储 API Key（`safeStorage`）

---

### 4. 自动更新机制

| 方案 | 工具 | 说明 |
|------|------|------|
| **内置更新** | electron-updater | 最成熟方案，支持差量更新 |
| **手动检查** | 自建服务器 | 简单，但体验一般 |

**推荐**：`electron-updater` + GitHub Releases

## 五、最终建议

### 如果追求快速上线 → Electron

**适用场景**：
- 希望 1-2 天内完成桌面版
- 不想学习新技术（Rust）
- 包体积不是首要考虑

**核心优势**：零重构，现有代码 100% 复用

---

### 如果追求极致轻量 → Tauri

**适用场景**：
- 有 Rust 学习意愿或已有基础
- 包体积敏感（需 < 50MB）
- 愿意投入 1-2 周开发时间
- 追求原生性能和体验

**核心优势**：包体积小，性能更好

---

### 不推荐 Neutralinojs

理由：生态不成熟，遇到问题难以解决，投入产出比不如前两者。
