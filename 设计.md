# Podcast2Markdown 设计文档

## 一、项目概述

**Podcast2Markdown** 是一款将播客音频智能转换为结构化文章的工具。用户输入播客链接或关键词搜索，平台自动完成音频转文字、文字整理成大纲文章、提取关键词和金句，最终输出 Markdown 格式的文档。

### 目标用户
- 播客听众：收藏和整理喜欢的播客内容
- 内容创作者：快速将播客内容二次创作为文章
- 学习者：将音频内容转化为可检索的笔记

---

## 二、核心功能设计

### 2.1 本地音频上传

**功能描述**：用户直接上传本地音频文件（MP3、WAV、M4A 等格式）。

**实现方案**：
1. 前端支持拖拽或选择文件上传
2. 后端接收文件并存储到临时目录或云存储
3. 解析音频文件获取基本信息（时长、格式等）
4. 调用转录服务

**数据模型**：
```typescript
interface AudioFile {
  id: string;
  fileName: string;
  fileSize: number; // bytes
  duration: number; // 秒
  format: string;
  uploadTime: string;
}
```

---

### 2.2 播客搜索与解析（V2）

**功能描述**：通过关键词搜索小宇宙播客，获取音频文件直链。

**实现方案**：
1. 抓取小宇宙网页版搜索结果（`https://www.xiaoyoyu.com/search`）
2. 解析搜索结果获取播客标题、作者、封面、音频链接
3. 提供音频链接后，调用转录服务

**数据模型**：
```typescript
interface Podcast {
  id: string;
  title: string;
  author: string;
  coverUrl: string;
  audioUrl: string;
  duration: number; // 秒
  publishDate: string;
}
```

---

### 2.3 音频转录

**功能描述**：将播客音频转换为文字。

**实现方案**：
1. 使用 **whisper.cpp** 本地推理 + Node.js 调用
2. 支持 **CUDA GPU 加速**（NVIDIA 显卡）
3. 返回带时间戳的字幕数据

**技术选型**：
- whisper.cpp：C++ 实现的高效 Whisper 推理，支持 CUDA
- whisper-node：Node.js 绑定，直接在 Node 进程中调用
- 模型：large-v3 或 medium（根据显存选择）

**成本优化策略**：
- 完全本地化，无 API 调用费用
- CUDA 加速大幅提升转录速度
- 支持 GPU 批量处理

**数据模型**：
```typescript
interface Transcript {
  podcastId: string;
  language: string;
  segments: TranscriptSegment[];
  fullText: string;
}

interface TranscriptSegment {
  start: number;
  end: number;
  text: string;
}
```

---

### 2.4 文字整理与结构化

**功能描述**：将转录文字整理成带大纲和结构的 Markdown 文章。

**实现方案**：
1. 调用 **Minimax（minimax2.7）** 进行内容理解
2. 提示词设计：
   - 识别主题和核心观点
   - 按逻辑章节组织内容
   - 生成标题层级（H1/H2/H3）
   - 保留关键引述和金句

**输出格式**：
```markdown
# 播客标题

> 嘉宾/来源信息 | 时长：XX分钟

## 大纲
- [章节1](#章节1)
- [章节2](#章节2)

## 正文

### 章节1
内容...

### 章节2
内容...
```

---

### 2.5 关键词与金句提取

**功能描述**：从文章中提取 Tags 和高亮观点。

**实现方案**：
1. **Tags 提取**：使用 Minimax 分析主题标签（5-10个）
2. **金句提取**：识别观点鲜明、易于传播的短句（50字以内）
3. **摘要生成**：生成 200 字以内的内容摘要

**数据模型**：
```typescript
interface ExtractedContent {
  tags: string[];
  highlights: Highlight[];
  summary: string;
}

interface Highlight {
  text: string;
  context: string; // 前后各50字的上下文
}
```

---

### 2.6 用户界面（Web端）

**技术栈**：
- 前端：Next.js 14 + Tailwind CSS + shadcn/ui
- 后端：Next.js API Routes 或 FastAPI
- 状态管理：React Query / Zustand
- 部署：Vercel 或 Railway

**页面结构**：

```
/                    首页 - 搜索入口
/search              搜索结果页
/episode/[id]        播客详情 + 转录状态
/result/[id]         生成的Markdown结果
/history             历史记录
```

**核心交互流程**：
1. 用户上传本地音频文件 → 解析文件信息
2. 开始转录 → 实时显示进度
3. 转录完成 → 自动生成文章
4. 预览/编辑 → 下载 Markdown

## 三、技术架构

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│   前端 Web  │────▶│   Next.js    │────▶│ whisper.cpp │
│  (Next.js)  │     │  API Routes  │     │   (CUDA)    │
└─────────────┘     └──────────────┘     └─────────────┘
                           │
                           ▼
                    ┌──────────────┐
                    │   Minimax    │
                    │ (minimax2.7) │
                    │  (整理/提取)  │
                    └──────────────┘
                           │
                           ▼
                    ┌──────────────┐
                    │   数据库     │
                    │  (SQLite/    │
                    │   PostgreSQL)│
                    └──────────────┘
```

**技术选型说明**：
- whisper.cpp：本地 C++ 实现，CUDA GPU 加速，无 API 费用
- Next.js API Routes：内置于 Next.js，简化部署
- SQLite：开发阶段使用，生产可切换 PostgreSQL

## 四、API 设计

### 4.1 上传音频文件
```
POST /api/upload
Body: multipart/form-data (file: audio)
Response: { audioId: string, fileName: string, duration: number }
```

### 4.2 提交转录任务
```
POST /api/transcribe
Body: { audioId: string }
Response: { taskId: string }
```

### 4.3 查询任务状态
```
GET /api/task/:taskId
Response: { status: 'pending' | 'processing' | 'completed' | 'failed', progress: number }
```

### 4.4 生成文章
```
POST /api/generate
Body: { transcriptId: string }
Response: { article: Article, extracted: ExtractedContent }
```

### 4.5 导出 Markdown
```
GET /api/export/:articleId
Response: markdown string
```

### 4.6 搜索播客（V2）
```
GET /api/search?keyword=xxx
Response: { podcasts: Podcast[] }
```

## 六、待解决问题

1. **长文本处理**：单次 Minimax 调用有 token 限制，需分段处理
2. **小宇宙反爬策略**（V2）：需要验证搜索接口的可用性，可能需要模拟请求或使用官方 API
3. **whisper.cpp CUDA 兼容性**：首次部署需确认 NVIDIA 驱动和 CUDA 版本
